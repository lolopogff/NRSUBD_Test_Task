# Чат сопровождения правовых запросов

MVP веб-приложения "Чат сопровождения правового запроса" для тестового задания.

## Бизнес-идея

Приложение закрывает разрыв между поиском правовой информации и фиксацией решения по кейсу.
Пользователь создает правовой запрос (кейс) и ведет по нему переписку, сохраняя контекст обсуждения в одном месте.

### Какую проблему решает продукт

В типичном процессе работы с правовым вопросом коммуникация распределена по нескольким каналам:

- поиск и аналитика — в справочно-правовой системе;
- финальные выводы — в отдельных документах.

Из-за этого теряется контекст:

- сложно восстановить, почему было принято конкретное решение;
- часть аргументов и ссылок на документы остается в личной переписке;
- при передаче задачи другому специалисту приходится собирать историю вручную;
- нет единой точки контроля статуса кейса.

### Почему такой чат нужен бизнесу

Чат сопровождения правового запроса формирует единое рабочее пространство по каждому кейсу:

- обсуждение, статус и история решений находятся в одной сущности (кейсе);
- специалист видит очередь вопросов и может оперативно менять статус;
- пользователь отслеживает прогресс без повторных обращений в разные каналы;
- команда получает накопление повторно используемой экспертизы (история кейсов).

### Бизнес-ценность для компании

- Снижение операционных потерь: меньше дублирующих запросов и пересылок контекста.
- Прозрачность процессов: в любой момент понятно, что в работе, что завершено и кто отвечает.
- Повышение качества сервиса: быстрые ответы и предсказуемый процесс для клиента.
- Масштабируемость поддержки: проще подключать новых специалистов и перераспределять нагрузку.

### Бизнес-ценность для пользователя

- Единая точка коммуникации по конкретной правовой ситуации.
- История обсуждения и решения не теряется во внешних чатах.
- Быстрая обратная связь от специалиста и понятный статус кейса.
- Удобный возврат к ранее решенным вопросам.

### Потенциал развития

- Назначение ответственного специалиста на кейс.
- Поиск по кейсам и автоподсказки по похожим обращениям.
- Привязка нормативных документов и шаблонов решений к переписке.
- Аналитика повторяющихся проблем для улучшения базы знаний и сервиса.

## Что реализовано

- Полноценные аккаунты: имя, username, пароль, роль.
- Регистрация и вход.
- Список правовых запросов.
- Создание нового правового запроса.
- Выбор запроса и просмотр истории сообщений.
- Отправка сообщений.
- Редактирование и удаление сообщений только автором.
- Мягкое удаление сообщения: в чате остается отметка "Сообщение удалено пользователем".
- Удаление кейса (владельцем или специалистом).
- Подгрузка истории сообщений (до 100 последних).
- WebSocket-обновления новых сообщений в реальном времени.
- Периодическая синхронизация сообщений как fallback.
- Разграничение доступа по ролям:
  - пользователь видит только свои кейсы и не может менять статус;
  - специалист видит все кейсы и может менять их статусы.

## Стек

- Frontend: React, Material UI, Redux Toolkit, TypeScript, React Create App.
- Backend: Node.js, Express, TypeScript.
- Хранилище: PostgreSQL.

## Структура

- `client` — frontend.
- `server` — backend API.

## Запуск

### Быстрый запуск одной командой (Docker Compose)

В корне уже задано фиксированное имя проекта Compose (`COMPOSE_PROJECT_NAME=legalchat` в `.env`), чтобы избежать ошибок на путях с кириллицей.

```bash
docker compose up --build
```

После старта:

- Frontend: `http://localhost:3000`
- Backend: `http://localhost:4000`
- PostgreSQL: `localhost:5432`

Остановка:

```bash
docker compose down
```

Остановка с удалением тома базы:

```bash
docker compose down -v
```

### 1) Backend

Перед запуском нужен PostgreSQL и база `legal_chat`.

Пример создания базы:

```bash
createdb legal_chat
```

Пример подключения по умолчанию:

```bash
postgres://postgres:postgres@localhost:5432/legal_chat
```

Можно переопределить через `DATABASE_URL` (см. `server/.env.example`).

Также задайте `JWT_SECRET` для подписи токенов.

```bash
cd server
npm install
npm run migrate
npm run dev
```

Сервер стартует на `http://localhost:4000`.

### 2) Frontend

```bash
cd client
npm install
npm start
```

Клиент стартует на `http://localhost:3000`.

Если backend запущен на другом адресе, укажите переменную:

```bash
REACT_APP_API_URL=http://localhost:4000
```

## API

- `POST /api/auth/register` — регистрация (`name`, `username`, `password`, `role`).
- `POST /api/auth/login` — вход (`username`, `password`).
- `GET /api/requests` — получить список запросов.
- `POST /api/requests` — создать запрос.
- `PATCH /api/requests/:requestId/status` — обновить статус запроса.
- `DELETE /api/requests/:requestId` — удалить кейс.
- `GET /api/requests/:requestId/messages?limit=100` — получить сообщения запроса.
- `POST /api/requests/:requestId/messages` — отправить сообщение.
- `PATCH /api/requests/:requestId/messages/:messageId` — редактировать сообщение.
- `DELETE /api/requests/:requestId/messages/:messageId` — удалить сообщение.
- `WS /ws?requestId=<id>&token=<jwt>` — подписка на новые сообщения.

Для защищенных HTTP API-запросов клиент передает:

- `Authorization: Bearer <jwt>`
